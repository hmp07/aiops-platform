## 一、核心理念

本平台的设计、开发与演进，始终遵循以下五条根本原则，所有功能需求的优先级、实现方式与取舍，都应对齐这些理念。

| 编号 | 核心理念        | 说明                                                         |
| ---- | --------------- | ------------------------------------------------------------ |
| P1   | 流程闭环        | 打通设备、IP、配置、巡检、告警、应用性能、日志、知识库和故障处理的完整链条，消灭信息孤岛 |
| P2   | 数据驱动        | 以自动采集的真实数据反向校准资产台账，实现"数实一致"         |
| P3   | AI 嵌入真实流程 | AI 不是孤立的大模型聊天窗口，而是融入监控、告警、巡检、排障等运维动作中 |
| P4   | 动态关联        | 运维数据反映现场实时状态，而非静态记录，支持实时拓扑与影响分析 |
| P5   | 人机协作        | AI 提供分析与建议，但生产环境的最终操作决策权与执行确认权始终在人 |

## 二、功能模块详细需求

### 模块 1：设备资产管理

**核心痛点**：设备台账数据不准确、维护成本高、无法反映真实在线状态。
**数据来源**：iTop CMDB（同步）+ 网络设备自动采集

| 编号 | 功能需求         | 优先级 | 说明                                                         |
| ---- | ---------------- | ------ | ------------------------------------------------------------ |
| A1.1 | 设备信息全量管理 | P0     | 管理设备名称、类型、位置、使用者、上线时间、所属系统/业务、资产生命周期状态（在用/备件/报废）等基础字段 |
| A1.2 | 设备属性扩展管理 | P0     | 支持 IP 地址、厂商、型号、序列号、软件版本、管理地址、机房/机柜、上联设备与端口等网络及基础架构专属字段的维护与展示 |
| A1.3 | 配置备份状态追踪 | P1     | 展示每台设备最近一次配置备份的状态（成功/失败/未备份）与时间 |
| A1.4 | 巡检状态追踪     | P1     | 展示每台设备最近一次巡检的结果概要（正常/异常/未巡检）与时间 |
| A1.5 | 资产自动校准     | P0     | 通过 SNMP/SSH 等方式自动采集设备型号、序列号、软件版本、板卡信息等，与 CMDB 记录对比，生成差异报告，**经人工确认后**可更新资产台账 |
| A1.6 | 批量导入/导出    | P1     | 支持通过 Excel/CSV 批量导入初始资产数据，支持按条件导出资产台账 |

**补充说明**：
- A1.5 的校准流程严格遵循人机协作原则：**自动采集→差异对比→人工审核→写入台账**，不直接自动覆盖生产资产记录。
- 校准过程中若发现"CMDB 有但采集不到"的资产，标记为"僵尸资产"；若发现"采集到但 CMDB 无"的设备，生成"幽灵设备"待处理工单。

### 模块 2：IP 地址管理（IPAM）

**核心痛点**：IP 使用状态不清、与设备实际占用不一致、子网/VLAN 缺乏统一管理。
**数据来源**：运维管理系统 + 网络设备 ARP 表/MAC 表采集

| 编号 | 功能需求        | 优先级 | 说明                                                         |
| ---- | --------------- | ------ | ------------------------------------------------------------ |
| B2.1 | IP 地址段管理   | P0     | 管理子网、VLAN、网关、地址段划分，支持 CIDR 标记             |
| B2.2 | IP 分配状态管理 | P0     | 每个 IP 标记为：已分配、空闲、保留；状态可手工修改，也可通过采集 ARP 信息自动更新 |
| B2.3 | IP 与设备关联   | P0     | 将 IP 地址与资产模块中的设备（及接口）关联，展示历史分配记录，支持反向查找：通过 IP 找设备 |

**闭环要求**：
- IP 状态变更（如从空闲变为已分配）需记录操作来源（手工指定 / 自动采集发现）。
- 与模块 1 设备资产双向联动：设备详情页可看到其所有 IP，IP 详情页可跳转到绑定设备。

### 模块 3：基础设施监控与告警

**核心痛点**：告警量大、运维人员易麻木；缺少自动证据收集和根因分析，排障依赖经验。
**数据来源**：Zabbix 监控系统 + 自有轻量采集探针

| 编号  | 功能需求         | 优先级 | 说明                                                         |
| ----- | ---------------- | ------ | ------------------------------------------------------------ |
| C3.1  | 设备在线状态监控 | P0     | 通过 Ping/SNMP 监测设备可达状态，不可达时触发告警            |
| C3.2  | 性能指标采集     | P0     | 采集 CPU、内存、磁盘、接口流量、光模块光功率、错误包计数等指标，对接 Zabbix 或自有采集 |
| C3.3  | 服务可用性监控   | P1     | 监控关键 TCP 端口、HTTP/HTTPS 接口、数据库连接等服务的可用性 |
| C3.4  | 告警规则配置     | P0     | 支持基于阈值、波动率、预测偏差等条件的告警规则，可灵活调整级别（提示/警告/严重/紧急）和触发条件 |
| C3.5  | 告警通知触达     | P0     | 支持企业微信、邮件、短信等多渠道通知，可按告警级别、设备分组设置通知策略 |
| C3.6  | 历史趋势可视化   | P1     | 提供性能指标历史趋势图表，支持自定义时间范围，可集成 Grafana 看板 |
| C3.7  | 告警自动降噪     | P0     | AI 自动识别重复、派生、瞬时抖动的冗余告警，进行聚合与抑制，减少告警风暴；展示降噪前后对比，可查看被抑制告警明细 |
| C3.8  | 告警自动取证     | P0     | 告警触发时，**作为证据收集执行引擎**，自动拉取该设备及关联设备的配置快照、关键日志片段、接口状态、邻居状态等原始数据，生成"取证快照"附加到告警上下文（与 H8.8 协同，详见下文职责边界说明） |
| C3.9  | 告警根因分析     | P0     | AI 综合取证信息、近期配置变更、历史同类告警处理经验，推断最可能原因，输出概率排序和关联证据链，供工程师确认 |
| C3.10 | 告警处理状态管理 | P0     | 提供告警处理的状态机：待认领→处理中→已解决→已关闭；支持添加处理备注、误报标记、转派、关联故障报告 |

**C3.8 与 H8.8 的职责边界**：

| 需求     | 定位                       | 职责                                                         | 触发时机                   |
| -------- | -------------------------- | ------------------------------------------------------------ | -------------------------- |
| C3.8     | 证据收集**执行引擎** | 按预置规则自动拉取原始数据（配置快照、日志片段、接口状态、邻居状态），不做加工整理 | 告警触发时，自动执行       |
| H8.8     | 证据**编排与上下文构建** | 将 C3.8 收集的原始证据按逻辑整理、关联、去重，构建可读的证据链，嵌入故障分析上下文，支撑 C3.9 根因分析 | 复杂故障场景，由 AI 调度   |

两者为上下游协作关系：**C3.8 负责"拿到什么"，H8.8 负责"怎么组织"**。

**流程闭环示例**：
`告警产生(C3.4) → 降噪聚合(C3.7) → 自动取证(C3.8) → 证据编排(H8.8) → AI 根因分析(C3.9) → 生成故障报告(草稿) → 人工认领处理(C3.10) → 处理方案总结(沉淀至知识库 G7.9) → 关闭告警`

### 模块 4：日志分析

**核心痛点**：日志量大、分散、查询门槛高，无法快速串联定位问题。
**数据来源**：日志管理系统（未来对接）+ 自采 Syslog/Agent

| 编号 | 功能需求            | 优先级 | 说明                                                         |
| ---- | ------------------- | ------ | ------------------------------------------------------------ |
| D4.1 | 多源日志汇聚        | P0     | 汇聚 Linux 系统日志、应用日志、网络设备日志、防火墙日志等，支持 Syslog、Filebeat、API 等方式接入 |
| D4.2 | 日志统一检索        | P0     | 提供关键字、时间范围、设备、日志级别等多维度的统一检索界面，降低使用门槛 |
| D4.3 | 日志可视化          | P1     | 提供日志量趋势、异常突增识别、错误分布等可视化图表           |
| D4.4 | AI 日志多维关联分析 | P0     | 将分散在不同日志源的记录按时间、设备、会话标识等维度串联为事件链，辅助发现因果关联，而非仅罗列日志 |
| D4.5 | 故障追溯            | P0     | 基于时间线还原故障发生前后的日志状态，标记关键事件节点，支持拖拽式时间范围选择，快速导出故障段日志 |
| D4.6 | 安全事件分析        | P1     | 针对防火墙日志、VPN 日志、登录失败等安全类日志提供预置分析规则和异常检测 |

**与模块 3、5 的联动**：
- 告警取证（C3.8）可自动从本模块拉取对应设备的近期日志。
- 日志时间线中可标记配置变更事件（E5.3），直观看到"变更 → 异常日志"的时间关联。

### 模块 5：配置备份与自动化

**核心需求**：配置的自动备份、变更追踪、合规对比。
**数据来源**：网络设备 SSH/Telnet/API

| 编号 | 功能需求         | 优先级 | 说明                                                         |
| ---- | ---------------- | ------ | ------------------------------------------------------------ |
| E5.1 | 设备配置自动备份 | P0     | 设备纳入资产管理后，自动触发首次配置备份，支持 SSH/Telnet/API 方式获取配置 |
| E5.2 | 配置定时备份     | P0     | 支持按日/周/月的周期性备份，可自定义备份策略；备份文件应加密存储 |
| E5.3 | 配置变更对比     | P0     | 与最近一次备份进行文本 diff，高亮差异行，记录变更时间、变更来源（自动/手工触发）、操作人（若可获取） |
| E5.4 | 配置回滚         | P1     | 支持从备份文件恢复设备配置（高危操作，需二次确认及权限校验） |
| E5.5 | 批量操作编排     | P1     | 支持通过 Ansible/Nornir/Netmiko 等框架，对多台设备批量执行命令或配置下发，带执行结果汇总 |
| E5.6 | 跨厂商适配       | P1     | 适配 Cisco、Huawei、H3C、Juniper 等主流网络设备，屏蔽命令行差异，提供统一的备份/比对接口 |

**与 AI 模块的职责边界**：
- E5.3 提供**确定性的文本差异计算**：输出"哪一行变了、变成什么"。
- 模块 8 的 H8.3（AI 配置差异分析）则基于此差异，**结合变更窗口、历史告警、知识库等信息，判断该变更属于正常计划内变更还是异常高危变更**，并给出风险评级。

### 模块 6：应用性能监控（APM）—— 对接 SigNoz

**核心痛点**：基础设施正常不代表应用正常；应用性能问题发现滞后、定界困难、缺乏跨层关联。
**数据来源**：SigNoz（基于 OpenTelemetry 的原生可观测性平台）

#### 6.1 SigNoz 简介

SigNoz 是一款基于 OpenTelemetry 的开源可观测性工具，支持 Metrics（指标）、Traces（链路追踪）、Logs（日志）三类遥测数据的统一采集、存储与展示，并内置异常检测与告警能力。在本平台中，SigNoz 作为**应用层的核心可观测性数据源**，与本平台的关系是：

- **SigNoz 负责**：应用性能数据的采集、存储、可视化、基础告警。
- **本平台负责**：将 SigNoz 的应用性能数据与基础设施监控数据**汇聚关联**，提供统一告警视图、跨层根因分析、AI 驱动的应用性能洞察。

#### 6.2 功能需求清单

| 编号  | 功能需求                | 优先级 | 说明                                                         |
| ----- | ----------------------- | ------ | ------------------------------------------------------------ |
| F6.1  | SigNoz 指标数据接入     | P0     | 通过 SigNoz API 接入应用 Metrics（请求量、延迟、错误率、吞吐量等），作为平台性能指标源之一，与基础设施指标并列展示 |
| F6.2  | SigNoz 链路追踪数据接入 | P0     | 接入 Trace 数据，支持在平台内查询调用链，将慢请求、错误请求与基础设施告警关联 |
| F6.3  | SigNoz 告警集成         | P0     | 接收 SigNoz 产生的应用性能告警（如 P95 延迟超阈值、错误率飙升），纳入平台统一告警流（模块 3），参与降噪、取证、根因分析 |
| F6.4  | 应用拓扑自动发现        | P0     | 基于 Trace 数据自动生成服务依赖拓扑图，展示服务间调用关系、请求量、平均延迟、错误率，支持钻取到单次调用详情 |
| F6.5  | 应用-基础设施关联映射   | P0     | 将应用服务与其运行所在的主机/容器/网络设备关联，建立"应用→容器→宿主机→交换机"的纵向拓扑，支撑跨层根因分析 |
| F6.6  | 应用性能趋势分析        | P1     | 提供应用级 SLA/SLO 看板（Apdex、P50/P95/P99 延迟、错误率趋势），支持按服务、端点、环境维度筛选与对比 |
| F6.7  | 应用异常检测            | P1     | AI 自动识别应用性能指标的异常波动（如凌晨低峰时段延迟突增），生成应用异常提示，区别于基础设施告警 |
| F6.8  | 应用日志关联            | P1     | 将 SigNoz 中的 Logs 数据与平台日志模块（模块 4）打通，支持从 Trace 一键跳转到关联日志，或在平台日志检索中同时查询应用日志 |
| F6.9  | 跨层数据关联            | P0     | **作为数据整合层**，建立应用服务与基础设施的关联数据集：将应用层告警（F6.3）与基础设施告警（C3.1-C3.3）、应用性能指标（F6.1）与基础设施指标（C3.2）、Trace 拓扑（F6.4）与 CMDB 设备拓扑（F6.5）进行数据对齐和关联，输出结构化的跨层关联数据，供 H8.10 消费 |
| F6.10 | 应用变更关联            | P1     | 将应用发布/部署事件与性能变化关联，当指标恶化时自动提示"最近 30 分钟内有发布记录"，辅助快速定位变更引入的问题 |

**F6.9 与 H8.10 的职责边界**：

| 需求   | 定位             | 职责                                                         | 输出                                                         |
| ------ | ---------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| F6.9   | **数据整合层**   | 汇聚应用层与基础设施层数据，进行时间对齐、实体关联（服务↔主机↔交换机），产出结构化的跨层关联数据集 | 关联数据集（如"交换机 X 上联宿主机 Y，宿主机 Y 运行服务 Z 的 3 个实例，当前实例错误率分别为…"） |
| H8.10  | **AI 推理引擎**  | 消费 F6.9 的关联数据，结合历史案例（G7.1）、配置变更（E5.3）等上下文，执行 AI 推理，判定故障边界并给出置信度 | 定界结论（如"75% 概率为数据库慢查询，20% 概率为网络丢包，5% 概率为应用代码问题"）+ 依据来源 |

两者为上下游协作关系：**F6.9 负责"数据长什么样"，H8.10 负责"结论是什么"**。

**核心区分原则（高内聚低耦合）**：
- **SigNoz** 是 APM 领域的专业工具，负责应用数据的**采集、存储、可视化、告警**，职责高度内聚。
- **AIOps 平台** 不试图替换 SigNoz，而是通过其 API **聚合数据**，将应用性能作为全局运维视图的一个维度，与基础设施、日志、配置变更等数据联合分析。
- **模块 3（基础设施监控）** 与 **模块 6（应用性能监控）** 在数据源和关注层次上清晰分离：模块 3 关注"网络/主机/OS 是否健康"，模块 6 关注"应用/服务/接口是否健康"。
- **模块 8（AI 智能运维）** 则同时消费两者的数据，实现跨层关联与智能分析。

#### 6.4 典型联动场景

| 场景                    | 数据链路                                                     | 价值                                                         |
| ----------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 业务延迟告警            | SigNoz 延迟告警 → 平台统一告警流 → 自动取证(C3.8) → 证据编排(H8.8) → 跨层数据关联(F6.9) → AI 跨层定界(H8.10) → 根因推断 | 秒级定界是应用自身慢、数据库慢、网络丢包还是基础设施过载     |
| 基础设施告警影响评估    | Zabbix 交换机告警 → AI 自动查询该交换机上联的宿主机 → 宿主机上运行的服务列表 → 关联服务的实时错误率 | 网络故障时，快速回答"影响哪些业务"，从"网络断了"升级到"交易成功率下降" |
| 应用拓扑 + 基础设施映射 | Trace 拓扑 + CMDB 设备信息 → 自动构建"服务→主机→交换机"三层拓扑 | 故障时一目了然：是东西向流量链路问题，还是南北向流量链路问题 |
| 发布回滚决策            | 应用部署事件 + 发布后 Apdex 趋势 → AI 给出"发布影响评估"，若 Apdex 恶化则建议回滚 | 用数据代替感觉做回滚决策                                     |

### 模块 8：AI 智能运维能力

**定位**：AI 不是独立的聊天窗口，而是嵌入到监控、巡检、配置管理、应用性能分析、排障等流程中的增强引擎。所有 AI 功能均依赖其他模块提供的高质量数据。

| 编号  | 功能需求                | 优先级 | 说明                                                         |
| ----- | ----------------------- | ------ | ------------------------------------------------------------ |
| H8.1  | AI 巡检报告自动生成     | P0     | 基于采集的性能数据、应用健康度、配置合规状态，自动生成巡检报告，用自然语言总结重点问题与需关注项 |
| H8.2  | 巡检提示词模板配置      | P0     | 支持自定义提示词模板，调整报告的关注维度（如侧重安全、性能、应用体验、老旧设备等）和输出格式 |
| H8.3  | AI 配置差异分析         | P0     | 接收模块 5（E5.3）的配置 diff 结果，结合变更时间窗口、历史变更模式、相关告警，判定变更风险等级（正常/可疑/高危）并解释理由 |
| H8.4  | AI 异常风险识别         | P0     | 从实时性能指标、日志异常、接口错误率、应用延迟等数据中，主动识别偏离基线的异常设备、链路或服务，生成风险提示工单 |
| H8.5  | AI 跨域关联分析         | P0     | 打通 IP、设备、接口、告警、配置变更、应用服务、调用链等多域数据，识别隐藏依赖关系，构建故障影响范围视图和连锁故障路径分析 |
| H8.6  | AI 排障经验自动沉淀     | P1     | 在故障处理过程中，记录工程师的排查步骤和决策依据，辅助生成排障经验草稿，减少 G7.9 的人工录入工作量 |
| H8.7  | AI 风险提示             | P1     | 在工程师执行高危操作（如配置回滚、批量修改、应用重启）前，基于当前设备/服务状态和历史操作记录，给出风险提示和回退建议 |
| H8.8  | AI 故障证据编排         | P0     | **作为证据编排与上下文构建引擎**，接收 C3.8 收集的原始证据，按逻辑整理、关联、去重，构建可读的证据链并嵌入故障分析上下文，为 C3.9 根因分析提供结构化输入（详见模块 3 的职责边界说明） |
| H8.9  | AI 问答（基于运维数据） | P1     | 支持自然语言问答，例如："支付服务最近1小时有异常吗？""接口 /api/order 的 P99 为什么突然变慢？""数据库慢查询影响了哪些业务？" |
| H8.10 | AI 跨层故障定界         | P0     | **作为 AI 推理引擎**，消费 F6.9 的结构化跨层关联数据，结合历史案例（G7.1）、配置变更（E5.3）等上下文，执行 AI 推理，判定故障边界（应用代码/数据库/网络/宿主机资源），输出定界结论与置信度（详见模块 6 的职责边界说明） |

**核心约束**：
- 所有 AI 建议均为"推荐"性质，禁止不经人工确认直接下发配置或执行高危操作。
- AI 分析的可解释性要求：每个结论需附带依据来源（如"该结论基于 E5.3 的变更记录、F6.9 的跨层关联数据和 G7.1 的 3 个历史案例"）。

## 五、非功能性需求

### 5.1 数据质量与异常处理

- **采集失败处理**：设备不可达时，重试 3 次，间隔 30 秒；仍失败则标记为"采集异常"，并记录最后可达时间。SigNoz 数据源不可用时，在页面上显示数据状态为"降级"，不影响其他模块功能。
- **脏数据兼容**：从 CMDB 同步的字段若为空或格式非法（如 IP 格式错误），系统应记录异常日志，不得直接写入正式资产表，需在"数据质量"面板展示待处理项。
- **同步延迟**：资产校准同步频率默认每小时一次，可配置；IP 状态基于 ARP 的更新频率默认 15 分钟一次；SigNoz 指标同步频率默认 1 分钟一次。
- **数据一致性**：当 CMDB 中设备被删除，平台应标记资产为"外部已删除"而非自动删除，由人工决定处理。

### 5.2 性能与规模

- **目标管理规模**：首期支持 5000 台网络设备，100,000 个 IP 地址的管理，200 个应用服务的监控。
- **告警处理能力**：高峰期支持每秒 200 条告警的接收与降噪处理（含 SigNoz 告警），降噪延迟不超过 5 秒（基于规则引擎；若涉及 AI 模型推理，延迟放宽至 30 秒）。
- **日志吞吐**：日志接收能力不低于每日 500GB，关键字检索响应时间 P95 < 3 秒（在推荐硬件配置下）。
- **Trace 查询**：从平台侧代理查询 SigNoz Trace 详情，响应时间 P95 < 2 秒（不含 SigNoz 自身处理延迟）。
- **并发用户**：日常支持 50 个并发操作，峰值 200 个并发查询不出现明显卡顿。

### 5.3 安全需求

- **传输加密**：所有对外服务接口强制 HTTPS/TLS 1.2+，内部服务间通信根据部署环境评估是否启用 mTLS。
- **配置备份加密**：所有设备配置文件在存储时必须加密（AES-256），备份下载需权限审批。配置 diff 时在内存中解密后计算，不落盘明文。
- **API 安全**：所有 API 必须基于 Token 或 OAuth2 鉴权，敏感接口（如配置回滚）需额外二次认证。
- **外部系统认证**：对接 Zabbix、iTop、SigNoz 等系统时，使用专用只读服务账号，密码/Token 加密存储，定期轮换。
- **审计日志保护**：操作审计日志存储于独立只写数据库或日志系统，禁止任何角色删除或修改。
- **敏感信息脱敏**：日志展示和配置文件中如含密码、密钥、Token 等敏感字段，自动脱敏处理后再呈现。

### 5.4 可用性与部署

- **自身监控**：平台关键服务（任务调度、数据采集、API 网关）需有健康检查接口，异常时自动告警通知平台管理员。
- **备份恢复**：平台自身数据库需定期备份，提供灾难恢复流程文档。

## 六、分阶段交付建议（MVP 切片）

基于敏捷思想，将功能按业务价值和技术依赖关系分三个阶段交付，每个阶段都能交付可验收的业务价值。

### MVP-1：基础资产与监控联动（核心闭环）

**目标**：打通资产、IP、基础设施监控、配置备份、告警处理的基础流程，建立"数实一致"的数据底座。
**包含功能**：
- 设备资产管理 A1.1, A1.2, A1.5（校准仅出报告）
- IP 地址管理 B2.1 ~ B2.3
- 设备在线监控 C3.1, 性能采集 C3.2, 告警规则 C3.4, 通知 C3.5
- 告警自动降噪 C3.7（首期可用简单规则，AI 降噪后续迭代）
- **告警处理状态管理 C3.10**（告警闭环的基础，必须首期交付）
- 配置自动备份 E5.1, E5.2
- 任务调度引擎 I9.4
- 基础权限 I9.1（最小实现：管理员 + 普通用户两个角色）

**验收价值**：能够自动发现并监控设备，资产台账开始被采集数据校准，IP 状态可视化，告警噪声下降，告警处理形成管理闭环。

### MVP-2：AI 智能分析与应用性能集成（AI 价值初现 + 应用层打通）

**目标**：引入 AI 巡检、配置变更智能分析、告警自动取证与证据编排，同时打通 SigNoz 应用性能数据，初步构建跨层视图。
**包含功能**：
- 配置变更对比 E5.3 + AI 配置差异分析 H8.3
- AI 巡检报告生成 H8.1, H8.2
- 告警自动取证 C3.8 + AI 故障证据编排 H8.8
- 历史趋势可视化 C3.6
- SigNoz 指标接入 F6.1
- SigNoz 告警集成 F6.3
- 应用拓扑自动发现 F6.4
- 应用-基础设施关联映射 F6.5
- 跨层数据关联 F6.9（数据整合层）
- AI 跨层故障定界 H8.10（初步版本，消费 F6.9 数据输出定界结论）

**验收价值**：每周可输出一份自动化巡检报告（涵盖基础设施与应用）；配置变更有了 AI 风险判断；告警具备自动取证和证据编排能力；应用性能数据首次进入运维统一视图，可看到"服务→主机→交换机"的关联拓扑，并具备初步的跨层故障定界能力。

### MVP-3：完整 AI 闭环与知识沉淀（终极形态）

**目标**：实现从告警到根因分析再到知识沉淀的完整 AI 驱动流程，日志分析能力上线，知识库形成自动化沉淀机制。
**包含功能**：
- AI 根因分析 C3.9
- AI 跨域关联分析 H8.5
- SigNoz 链路追踪接入 F6.2
- 应用性能趋势分析 F6.6, F6.7
- 知识库核心功能 G7.1, G7.2, G7.8, G7.9
- AI 问答 H8.9
- 日志分析模块 D4.1, D4.2, D4.4, D4.5
- AI 排障经验自动沉淀 H8.6
- AI 风险提示 H8.7

**验收价值**：典型故障自发现到给出根因分析报告，能自动定界是应用层问题还是基础设施问题；处理经验自动沉淀至知识库；复杂问题可通过自然语言查询获得辅助。
